# AirTerm 产品需求文档 (PRD)

## 产品概述

### 一句话定义

AirTerm 是一个让你在手机上远程监控和操控 Mac 上 Claude Code CLI 会话的工具。

### 目标用户

使用 Claude Code CLI 的开发者，日常场景：

- 在 Mac 上同时跑多个 Claude Code 会话处理不同任务
- 离开工位时想知道任务完成了没、是否卡在确认环节
- 想用手机快速给 Claude 发一条追加指令，不必回到电脑前
- 想在一个地方统一管理所有会话，而非在多个终端窗口间切换

### 核心价值

| 痛点                      | AirTerm 的解决方式          |
| ------------------------- | --------------------------- |
| 离开电脑就不知道任务状态  | 手机实时查看所有会话输出    |
| Claude 卡在权限确认，白等 | 手机推送通知，一键允许/拒绝 |
| 想追加指令要回到电脑前    | 手机直接输入，等于远程键盘  |
| 多个终端窗口难以管理      | 统一面板，一目了然          |

| 角色            | 说明                                                        |

## 功能需求

### P0：核心功能（MVP 必须）

#### F1: 内置终端 + 外部终端监控

**描述：** Mac 应用同时支持两种使用方式：在 AirTerm 内置终端中启动 CLI（体验与 iTerm2 完全一致），以及在后台自动发现并监控用户在外部终端中已经运行的 CLI 会话。两种方式同等重要，并列呈现。

**用户故事：**

> 作为开发者，我可以在 AirTerm 里直接开终端跑 claude，也可以在 iTerm2 里照常工作——AirTerm 在后台自动发现并监控。两种方式的会话统一管理，手机端看不出区别。

**验收标准：**

子进程模式（内置终端）：

- [ ] 内置终端面板，支持 pty + 完整终端体验
- [ ] 可启动 claude CLI 及任意命令
- [ ] 通过 `--input-format stream-json` 获取结构化事件
- [ ] 无需辅助功能权限
- [ ] CPU 占用近零（事件驱动）

Accessibility 模式（外部终端监控）：

- [ ] 自动检测外部终端中的 `claude` 进程
- [ ] 识别每个进程对应的终端应用（Terminal.app / iTerm2 / Warp）
- [ ] 识别每个会话的工作目录
- [ ] 新启动的 claude 进程在 5 秒内被发现
- [ ] 进程退出后在 5 秒内标记为已结束
- [ ] 需要辅助功能权限

#### F2: 终端内容读取

**描述：** 读取终端输出内容并推送到手机。子进程模式通过 pty 直接获取；AX API 模式通过 Accessibility API 读取外部终端窗口。

**用户故事：**

> 作为开发者，我希望在手机上看到 Claude Code 的实时输出，就像看着我 Mac 的屏幕一样。

**验收标准：**

子进程模式：

- [ ] pty 直接捕获终端输出流
- [ ] `--input-format stream-json` 提供结构化事件（diff、工具调用、确认请求）
- [ ] 输出变化实时推送，无需轮询

Accessibility 模式：

- [ ] 能读取 Terminal.app 窗口文本
- [ ] 能读取 iTerm2 窗口文本
- [ ] 内容变化在 1 秒内检测到
- [ ] 支持多个 Tab/Pane 分别读取
- [ ] 正确处理 ANSI 颜色代码
- [ ] 首次运行引导用户授予辅助功能权限

#### F3: 远程输入

**描述：** 从手机发送文字或按键到 Mac 上指定的终端窗口。

**用户故事：**

> 作为开发者，我在手机上输入一段指令后点发送，Claude Code 应该收到这段输入并开始执行，就像我在 Mac 上打字一样。

**验收标准：**

- [ ] 文本输入后自动加回车发送
- [ ] 支持发送 `y` / `n` 确认
- [ ] 支持发送 Claude Code 斜杠命令（`/commit` 等）
- [ ] 输入到达 Mac 终端的延迟 < 500ms
- [ ] 发送后有视觉反馈（发送中 → 已送达）
- [ ] 高危命令在 Mac 端弹窗确认

#### F4: 混合连接 (内网优先 + 外网回退)

**描述：** Mac 和手机优先通过内网直连通信，外网中继服务器作为热备回退。切换无感知。

**用户故事：**

> 作为开发者，在家/办公室时 Mac 和手机走内网直连，延迟极低；出门在外时自动切到外网中继，我完全不用管网络切换的事。

**验收标准：**

- [ ] Mac 内嵌 WebSocket 服务器，监听内网
- [ ] 手机打开时并行尝试 LAN 和 WAN，LAN 优先
- [ ] LAN 不可用时自动回退到 WAN，切换 < 1 秒
- [ ] LAN 恢复后自动切回，用户无感知
- [ ] 双通道同时保活（LAN active + WAN standby）
- [ ] 切换过程中消息不丢失（seq/ack + 重传）
- [ ] LAN 连接通过 Challenge-Response 认证
- [ ] 配对时二维码包含 Mac 内网地址
- [ ] Mac IP 变化时通过 WAN 推送新地址

#### F4a: 外网中继服务器

**描述：** 消息转发服务，作为 WAN 通道。默认使用官方公共实例 `relay.airterm.dev`，用户也可自部署。

**用户故事：**

> 作为开发者，我安装 AirTerm 后开箱即用，不需要自己部署服务器。如果我对安全有更高要求或需要自主可控，可以自部署。

**验收标准：**

- [ ] 官方公共实例 `relay.airterm.dev` 开箱可用
- [ ] Mac App 默认连接公共实例，无需配置
- [ ] 设置页支持切换为自部署地址（高级选项）
- [ ] Mac 和手机通过 WebSocket 连接服务器
- [ ] 消息在 200ms 内转发
- [ ] 支持 Mac 离线重连，自动恢复
- [ ] 手机断线重连后能获取最新状态
- [ ] 服务器不存储、不解密任何消息内容
- [ ] Docker 一键部署
- [ ] 健康检查接口 `/health`

#### F5: 设备配对

**描述：** Mac 和手机通过扫描二维码完成一次性安全配对。

**用户故事：**

> 作为开发者，我在 Mac 上点击"配对新设备"，用手机扫码，几秒内就能开始使用，不需要注册账号或输入复杂信息。

**验收标准：**

- [ ] Mac 端生成二维码，包含服务器地址和配对信息
- [ ] 手机扫码后自动跳转 Web 页面并完成配对
- [ ] 配对码 5 分钟过期
- [ ] 配对码一次性使用
- [ ] 配对过程中交换端到端加密公钥
- [ ] 配对完成后手机自动进入控制台
- [ ] 无需注册账号

#### F6: 端到端加密

**描述：** 所有通信内容使用 E2E 加密，中继服务器无法解密。

**用户故事：**

> 作为开发者，我的代码和终端输出是敏感信息，即使中继服务器被入侵，攻击者也不应该能看到我的任何数据。

**验收标准：**

- [ ] 使用 X25519 密钥交换 + ChaCha20-Poly1305 加密
- [ ] 服务器仅转发密文
- [ ] 每条消息使用唯一 Nonce
- [ ] 私钥仅存储在用户设备本地
- [ ] Mac 端私钥存储在 macOS Keychain
- [ ] 手机端私钥使用 Web Crypto API non-extractable

### P1：重要功能（V1.0 目标）

#### F7: 智能输出解析

**描述：** 将 Claude Code CLI 的终端输出解析为结构化的富文本视图。

**用户故事：**

> 作为开发者，我不想在手机上看原始终端文本，我希望看到格式清晰的消息气泡、语法高亮的代码差异、可折叠的工具调用。

**验收标准：**

- [ ] 识别并渲染 Claude 的文字消息为聊天气泡
- [ ] 识别代码 diff 并以语法高亮展示（添加行绿色、删除行红色）
- [ ] 识别工具调用（Read、Edit、Bash 等）并渲染为可折叠卡片
- [ ] 识别权限确认请求并展示为醒目的确认操作栏
- [ ] Bash 命令输出可折叠展开
- [ ] 长输出自动折叠，显示摘要

#### F8: 确认操作栏

**描述：** 当 Claude Code 请求权限确认时，在手机上显示醒目的确认按钮。

**用户故事：**

> 作为开发者，Claude Code 请求执行 `npm test` 时，我在手机上看到一个确认框，点"允许"就行，不用回电脑按 y。

**验收标准：**

- [ ] 自动检测终端中的 `[y/n]` 或类似确认提示
- [ ] 显示确认内容和操作描述
- [ ] 提供"允许"和"拒绝"两个大按钮（48px 高）
- [ ] 点击按钮后有触觉反馈
- [ ] 需要确认的会话在列表中置顶并显示警告图标
- [ ] 确认动作 < 500ms 到达终端

#### F9: 推送通知

**描述：** 当会话需要确认或任务完成时，主动通知用户。

**用户故事：**

> 作为开发者，我不想一直盯着手机看任务进度，Claude 需要我操作时推送通知就行。

**验收标准：**

- [ ] 权限确认请求时推送通知
- [ ] 会话结束时推送通知
- [ ] 会话出错时推送通知
- [ ] 支持通过 Bark (iOS) 推送
- [ ] 支持通过 Telegram Bot 推送
- [ ] 可配置通知类型（全部 / 仅确认 / 关闭）

#### F10: 快捷指令面板

**描述：** 预设常用命令，一键发送。

**用户故事：**

> 作为开发者，我经常发 `/commit`、`/review` 或 `y` 确认，不想每次都打字。

**验收标准：**

- [ ] 内置常用命令：`/commit`、`/review`、`继续`、`y`、`n`
- [ ] 可自定义快捷指令
- [ ] 显示为可横向滚动的药丸按钮
- [ ] 点击后直接发送（不需要二次确认）
- [ ] 危险指令（如 `Ctrl+C`）标红显示

#### F11: Mac 菜单栏 UI

**描述：** Mac 端以菜单栏应用形式运行，点击图标显示会话概览。

**用户故事：**

> 作为开发者，AirTerm 应该安静地待在菜单栏，我点一下图标就能看到所有会话状态和配对设备。

**验收标准：**

- [ ] 菜单栏图标显示连接状态（已连接/离线）
- [ ] 点击显示下拉面板
- [ ] 面板中列出所有活跃会话及其状态摘要
- [ ] 提供"配对新设备"入口
- [ ] 提供"已配对设备管理"入口
- [ ] 提供"设置"入口
- [ ] 开机自启选项

## 用户流程

### 流程一：首次安装与配对

```
1. 用户在 Mac 上安装 AirTerm.app（Homebrew 或直接下载）
2. 首次启动 → 引导授予辅助功能权限（可跳过，后续随时开启）
3. 点击菜单栏图标 → "配对新设备" → Mac 显示二维码
4. 手机扫码 → 完成配对 + SAS 验证 → 进入会话列表
5. 提示"添加到主屏幕"（PWA）
```

**简化要点：**

- 默认连接公共中继 `relay.airterm.dev`，无需配置服务器地址
- 辅助功能权限引导**在首次启动中展示但不强制**——跳过后仍可使用内置终端模式，外部终端监控待授权后自动生效
- 配对流程从 9 步精简到 5 步

### 流程二：日常使用（同一网络）

```
1. Mac 上正常使用 Claude Code CLI（任意终端）
2. AirTerm 自动发现会话
3. 手机打开书签 → 自动内网直连 Mac（无需中继服务器）
4. 看到会话列表，延迟极低
5. 需要时输入指令或点快捷按钮
6. Claude 需要确认时 → 手机弹通知 → 点击允许
7. 任务完成 → 推送通知
```

### 流程三：离开工位（网络切换）

```
1. 开发者给 Claude Code 布置了一个大任务
2. 离开工位去开会/吃饭（离开 WiFi 范围）
3. 内网断开 → 自动切到外网中继（< 1 秒，用户无感）
4. 手机收到通知："auth 重构 需要确认：执行 npm test"
5. 打开 AirTerm Web → 查看上下文 → 点击"允许"
6. Claude 继续执行
7. 回到工位（重新连上 WiFi）→ 自动切回内网直连
8. 手机收到通知："auth 重构 已完成"
9. 回到座位，代码已经改好了
```

## MVP 范围定义

MVP 分两个阶段交付：

```
MVP 第一阶段 — 核心验证（明文通信，跑通链路）:
  ✅ F1  内置终端 + 外部终端监控（双模 AgentAdapter）
  ✅ F2  终端内容读取（pty 直读 + AX API）
  ✅ F3  远程输入
  ✅ F4a 外网中继服务器（公共实例 relay.airterm.dev）
  ✅ F5  设备配对（简化版，token 认证）
  ✅ F7  智能输出解析（基础版：消息 + diff + 确认）
  ✅ F8  确认操作栏
  ✅ F10 快捷指令面板
  ✅ F11 Mac 菜单栏 UI

MVP 第二阶段 — 安全加固:
  ✅ F6  端到端加密（E2EE）
  ✅ F5  设备配对（完整版：密钥交换 + SAS 验证）
  ✅ 安全修复（JWT 短期化、SQLCipher、速率限制等）

后续阶段:
  ⬜ F4  混合连接 LAN 直连（需原生移动壳）
  ⬜ F9  推送通知（需原生移动壳 APNs）
  ⬜ 原生移动壳（iOS/Android）
  ⬜ 多设备支持
  ⬜ 会话历史
  ⬜ Warp 支持
  ⬜ iPad 适配
```

## 技术约束

| 约束                            | 原因                                         |
| ------------------------------- | -------------------------------------------- |
| 内置终端 + pty 子进程模式       | 可靠的 CLI 集成方式，无需辅助功能权限        |
| AX API 外部终端监控模式         | 无侵入接管已有终端的唯一方式，需辅助功能权限 |
| 两种模式并列，不分主次          | 满足不同工作习惯，会话统一管理               |
| 必须端到端加密                  | 传输的是用户源代码，安全底线                 |
| 手机端必须是 Web（非原生 App）  | 降低使用门槛，扫码即用，无需安装             |
| 服务器必须零知识                | 用户可能用公共实例也可能自部署               |
| 默认使用公共中继实例            | 零配置开箱即用，降低使用门槛                 |
| Mac 应用必须是原生 Swift        | pty 管理和 Accessibility API 需要原生调用    |
| AgentAdapter 接口不绑定特定 CLI | 架构预留，未来可扩展到 Codex、Gemini CLI 等  |
