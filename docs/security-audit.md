# 安全审计报告

**审计日期:** 2026-03-31
**审计范围:** 架构设计文档、安全方案、隐私方案、通信协议
**整体评级:** MEDIUM（设计思路正确，存在需修复的具体问题）

---

## 核心结论

### 中继服务器能否获取用户隐私？

**自部署场景:** 基本安全。用户信任自己的服务器，E2E 加密有效。

**公共实例场景:** 存在根本性缺陷。前端 JS 代码由服务器托管，恶意服务器可注入后门绕过 E2E 加密。必须将前端独立部署后才能称为「零知识」。

### Claude 会话能否被劫持？

存在多个攻击向量：配对 MITM、消息重放、JWT 劫持、Accessibility API 越权。已制定修复方案，详见下方。

## 发现的问题

### CRITICAL

**C1: 前端与中继服务器同域托管**

- 风险: 恶意服务器注入修改后的 JS，窃取私钥或明文
- 修复: 前端独立部署到 Cloudflare Pages，二维码包含 JS 哈希用于验证

**C2: 配对阶段手机公钥经服务器中转**

- 风险: 经典 MITM，恶意服务器替换公钥后解密所有通信
- 修复: 配对完成后双方基于 shared secret 生成 SAS 安全码（4 位数字），用户目视确认

### HIGH

**H1: 无消息重放防护**

- 风险: 截获加密的 "allow" 消息可在未来重放
- 修复: 每个方向递增序列号，纳入 AEAD 的 AAD

**H2: .env.example 缺少 DB_ENCRYPTION_KEY**

- 风险: 部署者遗漏配置，数据库明文存储
- 修复: 补全变量 + 服务器启动校验

**H3: SQLCipher pragma 字符串拼接**

- 风险: 环境变量含特殊字符时可能注入
- 修复: 参数化 pragma 调用

**H4: Accessibility API 权限无边界**

- 风险: 可读写任何窗口，不限于终端
- 修复: 操作前验证目标窗口 bundle ID 白名单

**H5: JWT 安全配置不完整**

- 风险: 30 天有效期 + 无吊销 = 长窗口暴露
- 修复: access token 15 分钟 + refresh token + 服务器黑名单

### MEDIUM

**M1: 危险命令拦截可被绕过** — 远程输入一律 Mac 弹窗确认
**M2: 心跳消息未加密** — 改走 E2E 信封
**M3: 消息大小侧信道** — padding 到 1KB 倍数
**M4: 配对码熵不足** — 8 位 alphanumeric + 3 次失败作废
**M5: Web 密钥存储依赖浏览器** — 严格 CSP + SRI

### LOW

**L1: 无前向保密** — 定期密钥轮换
**L2: 审计日志无防篡改** — hash chain
**L3: 重连无 certificate pinning** — 记住首次证书

## 值得肯定的设计

- X25519 + ChaCha20-Poly1305 是正确的现代加密组合
- Mac Keychain `WhenUnlockedThisDeviceOnly` 级别正确
- Web Crypto `non-extractable` 是浏览器端最佳实践
- 服务器仅内存转发，不缓存密文
- 配对码一次性使用 + 5 分钟过期
- 分层防御思路（传输加密 + E2E + 设备认证 + 命令过滤 + 数据库加密）
